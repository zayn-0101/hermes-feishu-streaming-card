# V3.4.0 Hermes Compatibility Design

## Goal

V3.4.0 makes the Feishu streaming card plugin compatible with both older supported Hermes releases and Hermes 0.13.0+ while fixing multi-profile routing, per-bot card titles, cron final delivery, attachment handling, card reply context, and issue #23.

## Scope

This release remains sidecar-only. Hermes keeps minimal marker-wrapped hooks that fail open, while card state, routing, rendering, Feishu API calls, diagnostics, and reply summary indexing remain in `hermes_feishu_card/`.

V3.4.0 is a releasable version. It must update version metadata, changelog, README files, config examples, testing docs, and migration guidance.

## Hermes Compatibility Matrix

`doctor` and `install` must select a hook strategy from version plus anchor detection:

| Hermes range | Strategy | Compatibility |
| --- | --- | --- |
| `v2026.4.23` through `0.12.x` | `legacy_gateway_run` | Existing V3.3-compatible gateway hooks |
| `0.13.0+` and current Hermes `main` | `gateway_run_013_plus` | New gateway hooks plus cron/reply/attachment capabilities when anchors exist |
| Below `v2026.4.23` or missing core anchors | unsupported | Refuse install |

`doctor` must show `hook_strategy`, anchor status, and `compatibility: full|partial|unsupported`. Missing core chat anchors blocks install. Missing optional cron, reply, or attachment anchors yields `partial` and a warning, but install can continue.

## Architecture

The existing modules keep their boundaries:

- `install/detect.py`: detect Hermes version, anchors, hook strategy, and optional capability support.
- `install/patcher.py`: apply/remove strategy-specific marker-wrapped patches.
- `hook_runtime.py`: extract locals, infer profile, extract reply/quote and attachment metadata, build events, and fail open.
- `server.py`: own sessions, locked route/card config, Feishu message ids, reply summary index, metrics, and diagnostics.
- `config.py`, `bots.py`, `runner.py`: load and resolve profile/bot/card configuration.
- `render.py`: render title, final text, footer, and attachment summaries into CardKit JSON.

The hook strategy is explicit. V3.4.0 must not grow a single fragile text patch that tries to cover all Hermes versions.

## Configuration

Card title resolution must support per-bot and per-profile defaults:

```yaml
card:
  title: Hermes Agent
  footer_fields: [duration, model, input_tokens, output_tokens, context]
  attachment_summary: true

profiles:
  work:
    card:
      title: Work Hermes
    feishu:
      app_id: cli_xxx
      app_secret: xxx
    bots:
      default: default
      items:
        default:
          app_id: cli_xxx
          app_secret: xxx
          card:
            title: Work Bot
        qa:
          app_id: cli_yyy
          app_secret: yyy
          card:
            title: QA Bot
```

Title priority is fixed:

1. `bot.card.title`
2. `profile.card.title`
3. top-level `card.title`
4. `Hermes Agent`

The route and resolved card config are locked on `message.started`. Later deltas and completion events must update the same card with the same bot, title, and footer settings.

## Profile Routing

Profile id extraction priority is:

1. `HERMES_FEISHU_CARD_PROFILE_ID`
2. Hermes locals/source/session/profile fields
3. `HERMES_HOME` or config path inference
4. `default`

Events must include stable `data.profile_id`. `/health` diagnostics must expose recent profile ids, inference source, event counts, and route errors.

This directly addresses issue #23: two Hermes profiles running independent gateways, two Feishu bots, and a shared patched `gateway/run.py` must route each profile to its own configured bot instead of falling back to only the default profile.

## Event Protocol

Keep `schema_version: "1"` and extend `data` for backward compatibility:

```json
{
  "profile_id": "work",
  "delivery_kind": "chat",
  "attachments": [
    {"kind": "file", "name": "report.pdf", "summary": "report.pdf"}
  ],
  "reply_to_message_id": "om_xxx",
  "reply_to_card_summary": "previous card answer summary"
}
```

Supported `delivery_kind` values are `chat` and `cron`.

## Cron Delivery

V3.4.0 covers cron final delivery only. It does not stream cron thinking, tools, or answer deltas.

When a Hermes cron job delivers a final result to Feishu, the plugin should send a completed card using the same profile/bot/title resolution rules. If card delivery fails, Hermes cron must fall back to native text delivery.

## Attachments And Media

Cards carry the text answer and a short attachment summary. Actual files, images, audio, video, and other media must remain on the Hermes native Feishu adapter path.

Completion suppression must not swallow native attachment delivery. For pure text Feishu replies, a successful completed card can suppress the native gray text. For replies containing attachments or media, the hook must allow the native attachment path to continue.

## Card Reply Context

The sidecar maintains an in-memory `feishu_message_id -> card summary` index after completed card updates. Hook/runtime attempts to extract `reply_to_message_id` or quote id from Hermes locals/source/event/message.

When a user replies to a card:

1. The hook extracts the quoted/replied message id.
2. It queries the sidecar summary index.
3. If found, the card summary is injected into event metadata or user-message context so Hermes can see what card was replied to.
4. If lookup fails or sidecar is unavailable, the message continues without injected card context.

The index is intentionally process-local in V3.4.0. Sidecar restart loses old summaries and degrades to native behavior.

## Error Handling

Install time is fail-closed:

- Core chat anchor missing: `install` refuses.
- Optional cron/reply/attachment anchors missing: `doctor` reports partial capability; install can continue with warnings.
- Unknown version with complete anchors: default to refusal in V3.4.0; no `--force` path is required.
- Existing corrupt markers or changed backups/manifests must continue to refuse unsafe restore/uninstall.

Runtime is fail-open:

- Sidecar unavailable: Hermes native behavior continues.
- Profile inference failure: use `default` and record diagnostics.
- Route/bot error: do not suppress Hermes native reply.
- Cron card failure: fall back to native cron delivery.
- Reply summary lookup miss: no injected summary, no user-visible failure.
- Non-Feishu platforms: no card emission or native-response suppression.

## Health And Diagnostics

`/health` should include compatibility, profile, reply, and cron observability:

```json
{
  "compatibility": {"hook_strategy": "gateway_run_013_plus", "capabilities": {}},
  "profiles": {"work": {"events": 10, "last_profile_source": "env"}},
  "routing": {"last_route": {}, "last_route_error": ""},
  "reply_index": {"entries": 12, "last_lookup": {}},
  "cron": {"cards_sent": 3, "fallbacks": 0}
}
```

Diagnostics must not leak `app_secret`, tenant tokens, bearer tokens, or real private message text beyond short safe summaries.

## Tests

Unit coverage:

- `detect.py`: version matrix, anchor capabilities, strategy selection.
- `patcher.py`: legacy and 0.13+ apply/remove/idempotency/corrupt marker behavior.
- `hook_runtime.py`: profile inference, reply id extraction, attachment metadata extraction, non-Feishu skip behavior.
- `config.py`, `bots.py`, `runner.py`: card title priority and profile/bot config merging.
- `server.py`, `session.py`, `render.py`: locked route/card config, attachment summaries, reply summary index, cron completed cards.

Integration coverage:

- `hermes_v2026_4_23` fixture install/restore loop.
- `hermes_0_13_plus` fixture install/restore loop.
- Two profiles and two bots both reply through their own profile route.
- Cron final delivery sends a completed card and falls back on card failure.
- Attachment completion keeps native attachment delivery available.
- Replying to a cached card injects summary context; missing summary fails open.

Manual smoke:

- Real Feishu two profiles and two bots.
- Real Feishu cron delivery.
- Reply to a completed card and confirm the model sees the card summary.
- Send image/file/media output and confirm the card has a summary while media is sent separately.
