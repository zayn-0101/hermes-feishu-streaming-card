# V3.8 设计方案：飞书卡片体验与运维能力升级

日期：2026-06-30
状态：等待维护者 review
负责人：baileyh8

## 背景

V3.7.0 已经完成 Docker 安装适配，并延续了 V3.6 系列对 Hermes runtime 兼容性、诊断能力和中断稳定性的修复。V3.8 应该从“插件能安装、能跟上 Hermes 变化”，进一步升级到“飞书卡片就是 Hermes 在飞书里的稳定主界面”。

最近暴露出来的问题，本质上都指向同一个产品缺口：

- 思考过程 / reasoning 在 burst 流式场景下仍可能截字、漏字。
- 卡片更新速度可能落后于 Hermes 事件流；finish 后，积压内容可能溢出成普通消息。
- 长表格、代码块跨过 `MAIN_CONTENT_CHUNK_CHARS` 后，可能被拆断并渲染成 raw markdown。
- 工具调用、思考内容、最终回答、footer、fallback 状态都挤在一张卡里，阅读层级不够清晰。
- 排查问题仍然过度依赖人工本地检查和真实飞书手测。

已确认的 V3.8 发布策略是分阶段：

- V3.8.0：优先交付卡片体验和流式稳定性核心。
- V3.8.1：交付卡片内运维命令和诊断能力。
- V3.8.2：补齐维护体系、E2E 测试和开放扩展面。

## 目标

1. 让飞书卡片在长回答、多工具、多 reasoning 的 Hermes 任务中仍然可读。
2. 防止流式完成阶段的竞态导致丢字、截断 thinking，或重复输出普通 fallback 消息。
3. 让插件状态和 runtime 健康情况能直接在飞书里检查。
4. 让后续 Hermes 兼容性和飞书渲染回归能自动测试，而不是主要靠人工真机试。

## 非目标

- V3.8 不做 Docker runtime 编排；V3.7.0 已经覆盖 Docker 安装适配。
- 不重写整个 sidecar 或 installer 架构。
- 不引入破坏性配置格式；如果确实需要，必须提供迁移路径和 doctor 提示。
- 不在卡片、日志或测试产物中暴露私有凭据、token、chat ID。
- 不强制所有用户展示 reasoning；reasoning 展示仍然应该可配置。

## V3.8.0：卡片体验与流式稳定性

### 范围

V3.8.0 是用户最直观看到的核心版本。优先级是正确性和可读性，而不是新增命令。

计划改动：

- 增加统一辅助 timeline，用于展示 reasoning 和工具调用。
- 最终回答放在独立的主内容区域。
- 增加卡片安全的 markdown 分块逻辑，覆盖长表格、代码块和大段 markdown。
- 增加 flush / drain 层，用于合并 burst 更新、正确唤醒 event loop，并在完成前 drain pending 更新。
- 记录 pending update queue length、coalesce count、terminal drain latency、Feishu API latency。
- 增加元素 / 内容预算安全网，保证辅助内容先折叠，最终回答不被牺牲。
- 保留当前 fallback 行为，但让 fallback 决策变得可诊断。

### 用户体验模型

卡片阅读顺序应该是：

1. Header：Hermes Agent 当前状态。
2. 主回答：最终回答或正在流式输出的回答。
3. 辅助 timeline：可折叠的 reasoning / tool timeline。
4. 工具摘要：timeline 折叠时显示 compact counts / statuses。
5. Footer：耗时、模型、token / context 摘要、更新 / fallback 健康状态。

辅助 timeline 按时间顺序展示：

- reasoning / thinking 内容开始时，创建一个 reasoning round。
- 回答文本或工具活动开始时，结束当前 reasoning round。
- 工具 start / update / complete 按发生顺序插入。
- 长工具参数和结果默认摘要展示，可以折叠。

### 流式保证

V3.8.0 的流式路径需要保证：

- 不因为 pending 卡片更新而丢弃 answer delta。
- `message.completed` 在关闭或 finalizing 卡片前，等待一个有边界的 final drain。
- final drain 期间如果飞书更新失败，卡片要记录可诊断的 fallback / partial 状态。
- burst stream 应该合并更新，而不是每个 token 都调用一次飞书 API。
- 除非卡片路径在 durable card message 创建前真正失败，否则完成阶段不应该再发送重复普通消息。

### Markdown 与卡片预算

渲染器应尽量避免在这些结构内部拆分：

- fenced code block
- markdown table
- list item
- inline code span

如果内容仍然超过卡片限制：

- 优先折叠旧的 reasoning / tool timeline 条目。
- 保留最终回答和最后的 error / fallback 状态。
- 宁愿展示可读的 partial card 和明确的“内容已折叠”提示，也不要输出损坏的 raw markdown。

### 验收标准

- DeepSeek-style burst stream 能保持 reasoning / answer 文本顺序完整。
- 超过旧 chunk 阈值的长表格能渲染为表格或可读 markdown，不出现 raw broken markdown。
- 单元测试覆盖 completion 前 drain pending updates。
- interrupt / abort 不再同时生成 streaming 和 non-streaming 两条最终答复。
- 相关 pytest 全部通过。
- 新增测试覆盖 burst flush、final drain、长表格 / 代码块分块、卡片预算折叠。

## V3.8.1：飞书内运维命令

### 范围

V3.8.1 增加一个轻量的飞书内命令入口，面向维护者和高级用户。具体命令名可以在实现阶段最终确认，设计目标如下：

- `/hfc help`
- `/hfc status`
- `/hfc doctor`
- `/hfc monitor`
- `/hfc reload`

命令前缀要避免和 Hermes 原生命令冲突，并在文档里写清楚。

### 命令行为

在当前 Hermes hook surface 允许的情况下，这些命令应该由插件在进入 LLM 前处理。如果 Hermes 正在执行请求且命令无法安全处理，插件应该回复一张简短状态卡，说明用户应该等待当前任务完成，还是先 stop。

预期卡片：

- `status`：插件版本、Hermes 版本 / 支持范围、runtime Python path、配置来源、bot 身份状态。
- `doctor`：凭据存在性、hook importability、sidecar reachability、卡片发送 / 更新能力、Hermes venv 路径。
- `monitor`：update queue length、coalesce count、Feishu latency、fallback counts、recent error codes。
- `reload`：配置重新加载结果和变更字段，敏感信息必须脱敏。

### 验收标准

- 命令不会触发 LLM 回答。
- 敏感值被脱敏。
- 私聊和群聊都能返回可读卡片。
- `doctor` 复用现有诊断逻辑，不重复造一套检查。
- `monitor` 展示 V3.8.0 新增的 flush / card metrics。

## V3.8.2：维护体系与扩展面

### 范围

V3.8.2 把 V3.8 的卡片与运维能力沉淀成可维护体系。

计划改动：

- 增加 mock Feishu E2E，用于断言 card create / update / close / fallback。
- 增加可选 real Feishu E2E，通过明确的环境变量启用。
- real E2E 自动创建 anchor message，不再要求手动提供 message_id。
- 增加 `docs/AGENT_GUIDE.md`，覆盖 hook lifecycle、card lifecycle、fallback chain、test matrix、release flow、常见 issue triage。
- 增加 / 扩展 Hermes v0.17+ 和最新 Hermes release 的兼容性测试。
- 评估一个窄范围 public helper API，让外部工具能扩展，但不暴露不稳定内部实现。

### 验收标准

- mock E2E 能在没有飞书凭据的 CI 中运行。
- real E2E 能在本地通过专用测试 bot / chat 运行，不污染生产群。
- Agent guide 足够让未来维护者快速定位 hook、renderer、fallback、installer、release 相关代码。
- 兼容性测试明确记录覆盖的 Hermes 版本。
- release checklist 包含卡片体验、运维命令和 E2E 验证步骤。

## 实现提示

### 候选内部模块

最终文件名可以在实现阶段调整，但大致结构建议如下：

- `hermes_feishu_card/flush.py`：flush controller 和 final drain 行为。
- `hermes_feishu_card/card_timeline.py`：reasoning / tool 时间线状态。
- `hermes_feishu_card/rendering.py`：卡片安全 markdown 分块和预算折叠。
- `hermes_feishu_card/metrics.py`：内存计数器和 latency snapshot。
- `hermes_feishu_card/commands.py`：飞书命令路由和命令卡片。
- `tests/e2e/`：mock 与可选 real Feishu E2E harness。

### 候选配置项

可能新增的配置字段：

- `flush_interval_ms`
- `final_drain_timeout_ms`
- `timeline_expanded`
- `max_timeline_items`
- `max_reasoning_chars`
- `max_tool_result_chars`
- `show_reasoning`
- `operations_command_prefix`

默认值要以安全为主，普通用户不需要调参也能受益。

### 迁移

现有用户升级到 V3.8.0 不应该需要编辑配置。新增配置应保持可选，并在 `doctor --explain` 中可见。如果旧的 env / config 名称仍有效，就继续兼容。

## 风险

- 飞书卡片限制不完全是 schema 限制，也包含行为限制。缓解方式：预算测试 + real E2E smoke。
- 更高频的卡片更新会增加 API 压力。缓解方式：coalescing、有界 flush interval、monitor counters。
- 命令拦截依赖 Hermes hook 可用性。缓解方式：优雅 fallback 和按版本启用。
- timeline 折叠可能隐藏有用调试细节。缓解方式：保留最近条目、计数和明确的 folded notice。
- real E2E 可能制造群聊噪音。缓解方式：专用测试群、env gate、尽量复用 anchor、短摘要。

## 待确认问题

- 最终命令前缀：`/hfc`、`/hermes-card`，或其他更适合用户理解的名称。
- V3.8.0 是否立即暴露 timeline folding 配置，还是先内部启用，到 V3.8.1 再开放。
- real E2E 文档定位为维护者专用，还是开放为高级用户验证方式。
- footer metrics 默认是否保持极简，详细指标只放在 monitor / doctor 卡片中。
